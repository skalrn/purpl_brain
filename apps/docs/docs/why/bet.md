# The Strategic Bet

## Agent cooperation vs. automatic interception

The most obvious approach to agent memory is to intercept sessions automatically — instrument the agent runtime, capture everything the agent outputs, and extract decisions without requiring the agent to do anything explicit. Some providers are pursuing this. It has appeal: it requires no developer instrumentation, no prompt changes, no compliance tracking.

purpl_brain does not do this. The write-back is cooperative: the agent is expected to call `brain_log_decision` or `POST /brain/agent-log` with a structured decision log. The agent must choose to write, and what it writes must meet quality standards (rationale present, alternatives considered, decisions not just implementation notes).

The bet is that cooperative write-back produces better data than automatic interception, and that the compliance gap can be closed with tooling rather than by removing developer choice.

Here is the reasoning. Automatic capture of session output extracts decisions from raw text generated by an agent that is not thinking about decision structure. The extraction is lossy — it finds what the agent said, not necessarily what the agent decided, and it cannot recover the alternatives that were considered but not verbalized. A cooperative write-back, where the agent is explicitly prompted to think about what it decided, why, and what alternatives it evaluated, produces structurally richer output. The rationale and alternatives fields are what distinguish a decision log from a session summary, and those fields require the agent's explicit attention.

The compliance cost is real. An agent that does not call the write-back tool produces no useful data for that session. The gap between cooperative compliance rates (60-90% depending on setup) and automatic capture (100%) is the price of richer structure. The bet is that 85-90% compliance with high-quality decision logs is more useful than 100% capture of low-quality summaries.

## Decision provenance beats conversational memory

Most agent memory systems store what agents said. purpl_brain stores what agents decided — specifically: what was chosen, what was rejected, why the choice was made, how confident the agent was, and what remains unresolved.

The difference is not cosmetic. Consider two memory entries for the same session:

**Conversational memory:** "The agent worked on auth-service and discussed JWT token expiry, considered refresh token rotation, and ultimately made some changes to the token handling code."

**Decision memory:** "Chose `node-jsonwebtoken` over `jose` because `jose@5.x` has a JWE incompatibility with the current token format (confidence: high). Rejected refresh token rotation pattern as unnecessary complexity at this stage (confidence: medium). Unresolved: session revocation strategy — needs decision before merge."

The second entry is what the next agent needs to not re-derive the library choice, not re-evaluate the refresh token pattern, and know exactly what question is waiting for an answer. The first entry tells the next agent that something happened with JWT tokens.

This distinction matters more as agent teams scale. A single agent on a single project can rebuild context from session summaries with enough effort. Five agents across three repos, running in parallel, with overnight sessions — the only thing that prevents contradictions and re-derivation at that scale is structured, queryable memory with explicit alternatives.

## Why this matters as agent teams grow

The current state of AI-assisted development is largely one-agent-per-developer, one-session-at-a-time. The trajectory is toward concurrent agents on the same codebase, autonomous overnight runs, and infrastructure agents making schema decisions alongside coding agents making API decisions.

At one agent per developer, the re-derivation cost is annoying but manageable. The developer is in the loop and can re-paste context or correct contradictions manually. At five concurrent agents across a codebase, the developer cannot be the human memory bus — there are too many sessions to supervise, and sessions happen while the developer is offline. The question is not whether there will be contradictions; it is whether there will be a system that detects them before they land in production.

The bet that purpl_brain makes is that this shift — from single-agent development to multi-agent development — happens within a two-to-three year window, and that the teams that establish shared memory infrastructure now will have a compounding advantage when it does. The brain that has been accumulating decision history for six months is more useful to an agent than an empty brain, and the team that has tuned their write-back compliance is more prepared for multi-agent coordination than the team that has been manually re-pasting context.

The strategic position: the memory layer should be built before it is urgently needed, because the cold-start problem is severe. A brain with six months of decisions is a different product than a brain that was just installed. The earlier a team installs and fills it, the more useful it becomes.
